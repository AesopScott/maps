Completed VA-023: Data-source classification and prompt masking

Claude's research output:
- Use a three-layer defense model: classification, masking, and scope enforcement.
- Classification should label data sources at intake, including PII, financial, health, business confidential, and public data. Unmarked data should fail closed or be treated as suspicious until labeled.
- Masking should happen before data enters models, tool calls, memory, logs, error messages, or observability streams. Useful transformations include tokenization, redaction, aggregation, and pseudonymization.
- Scope enforcement must apply consistently across prompt inputs, tool arguments/responses, memory/context storage, structured logs, debugging output, and final output assembly.
- Authorized unmasking should require cryptographic authorization proof, audit logging, and role-bound access. Observability sampling should be masking-aware and should not reintroduce sensitive data.
- Proof should include masking efficacy tests, injected-PII coverage audits, utility-retention tests, authorized/unathorized unmask validation, and prompt-injection red-team attempts to leak masked data.

Vik architecture-owner interpretation:
VA-023 is a core security substrate for every Mojo/Mindshare agent that touches customer data, memory, logs, or tools. The key architecture pattern is not one regex filter at prompt time; it is a source-bound data contract that classifies data before it enters the agent context and carries that classification through prompts, tool calls, memory writes, logs, and output rendering. This should become part of the agent runtime gate and MAPS deploy/evaluate criteria: no production or customer-facing agent should run without an explicit data classification and masking policy.

Productization recommendation:
Build as an enabling component first, with possible packaged product surface later. I do not recommend launching it as a standalone branded product by itself because buyers expect data masking as part of secure AI infrastructure. I do recommend making it a mandatory runtime component and a visible enterprise trust feature: Secure Agent Contexts. Over time it can support premium policy packs for healthcare, fintech, legal, and enterprise compliance, but its first job is to make every Mojo security-agent product safe enough to sell.
