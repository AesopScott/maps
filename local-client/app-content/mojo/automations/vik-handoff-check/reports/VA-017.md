Completed VA-017: Red team automation agents

Claude research output: red-team automation should use a three-layer isolation model: agent sandbox, approval gate, and oversight console. Agents can only use approved signed libraries, approved campaign scope, expiring campaign tokens, and audited action paths. Scope drift, token expiry, unapproved imports, or network escape must halt execution and log the violation.

Architecture pattern:
- Red Team Agent Sandbox restricts libraries, filesystem, network, subprocess behavior, and campaign scope.
- Approval Gate defines campaign scope, duration, target list, rules of engagement, multi-person signoff, and token expiry.
- Oversight Console captures real-time action logs, blocked attempts, kill-switch events, and post-action audit reports.
- Agent runtime validates signed library manifest at startup.
- Network access is limited to approved targets only.
- Every tool call is logged before execution.

Guardrails:
- Approved tools only through signed library bundles.
- Target scope binding through campaign target list and network ACL.
- Time-bound campaign tokens revoke automatically on expiry.
- Action quotas limit runaway execution.
- No lateral movement outside exhaustive target list.
- No exfiltration except findings returned to approval console.
- Config is immutable after approval.
- Approval withdrawal causes graceful halt.
- Scope violation causes shutdown and incident log.

Eval suite:
- Containment tests for unsigned library imports, lateral network moves, token expiry, and approval withdrawal.
- Scope integrity tests for out-of-range CIDR scans, DNS scope bypass, and time-window expiration.
- Action audit tests requiring one log entry per action before execution.
- Escape tests for malicious campaign JSON, subprocess fork, write outside campaign directory, and audit deletion.
- Operability tests for campaign approval, revocation, rollback visibility, and post-campaign audit.
- Confidence metrics: escape attempts caught, audit completeness, approval integrity.

My architecture-owner interpretation: VA-017 should not be framed as “autonomous red team.” It is an authorization and containment harness for adversarial evals. The useful productizable asset is not offensive capability; it is the proof that offensive automation stayed inside approved scope. That distinction matters legally and architecturally.

Productization recommendation: enabling component, internal first. Do not sell as standalone offensive automation. It may become part of an agent governance, security-lab, or red-team compliance platform after validated campaigns prove zero escapes and complete audit. Candidate framing: OpSec Harness or Red-Team Automation Governance, not attack automation.

Boundaries held: no implementation beyond report/state/backlog record, no Git/release, production, external communication, spending, secrets, authority expansion, runtime activation, or promotion.
