Completed VA-024: RBAC for AI apps and agents

Claude's research output:
- Agent RBAC should decouple identity from executable authority. A role title should not imply what an agent, user, workflow, tool, or memory store can actually do.
- Use three layers: identity layer, authority profile layer, and binding context.
- Identity layer names users and agents, but grants no authority by itself.
- Authority profiles define explicit capability sets for user-direct actions, delegated agent actions, workflow steps, tool access, memory access, and sub-agent delegation.
- Binding context defines where and when authority applies, with enforcement at tool invocation, memory access, workflow sequencing, and sub-agent spawning.
- Agent authority must never exceed the user authority plus explicit delegation bounds. User role grants zero agent permissions by default.
- Every agent should receive a signed authority manifest at spawn time. Revoking user authority should immediately revoke delegated agent authority.
- Guardrails should include isolated tool execution, memory ACLs, no wildcard memory access, explicit `can_delegate` flags, approved sub-agent lists, and audit logs for every grant, use, revoke, escalation, and denial.
- Proof should include manifest validation, multi-agent workflow simulations, adversarial privilege-escalation attempts, and audit reconstruction tests.

Vik architecture-owner interpretation:
VA-024 is central to Mojo's role-agent architecture. The critical principle is: title is not authority. A role name, maturity level, workflow label, or model identity cannot grant executable power. Authority must be explicit, scoped, bound to context, and enforced at runtime. For Mojo, this maps directly to role contracts, tool permissions, memory/RAG stores, Claude/Codex adapters, workflow queues, and future customer-facing agents. It also gives us a clean way to explain why a role may recommend broadly while an agent may execute narrowly.

Productization recommendation:
Build as an internal capability first and an enabling component second. I do not recommend a standalone branded RBAC product at the start because agent RBAC will become expected platform plumbing. I do recommend treating authority-binding as a mandatory core feature in every Mojo agent product and later exposing it as an SDK/API capability for customers building their own multi-agent systems. The product language should be authority-binding or agent permissions infrastructure, not generic RBAC.
