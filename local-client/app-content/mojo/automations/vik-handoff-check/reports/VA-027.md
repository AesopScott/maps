Completed VA-027: Guardrail and policy enforcement across prompts, outputs, and actions

Claude's research output:
- Guardrails should compose as a staged enforcement pipeline: input validation, prompt hardening, execution/tool gate, action approval, and output filtering.
- Each layer should operate independently while reporting to a unified policy engine that aggregates decisions.
- Design principles should include fail-safe defaults, layer independence, priority-based conflict resolution, observability, and deterministic fallback behavior.
- Input validation handles schema, length, format, content scanning, injection detection, jailbreak signatures, and rate limits.
- Prompt hardening protects system/user boundaries, scope declarations, and adversarial pattern detection.
- Tool restriction uses capability-based access, parameter validation, dependency constraints, and execution vetoes.
- Action approval gates sensitive data access, external calls, state mutation, and stale approval windows.
- Output filtering handles PII/secrets redaction, instruction leakage prevention, harmful content detection, and schema validation.
- Policy priority should separate hard policies, soft policies, and heuristics.
- Proof should include injection attacks, jailbreak corpora, tool abuse, output evasion, composability tests, false-positive impact, and latency checks.

Vik architecture-owner interpretation:
VA-027 should become Mojo's layered policy-enforcement pattern. The key is composition: no single guardrail is enough, and no guardrail layer should silently weaken another. For Mojo agents, every input, prompt, tool call, action, memory write, and output needs a policy decision with reason, confidence, owner, and audit trace. This aligns with strict-intent gates, recommendation/action separation, RBAC, scoped secrets, data masking, and runtime hardening.

Productization recommendation:
Build as an internal capability now and an enabling component after proof. I would not sell guardrails as a standalone external product yet; guardrails need deep runtime integration and customer-specific policy design. I would package the pattern inside Mojo's agent runtime first, then expose policy packs, middleware, eval harnesses, and audit views as part of a broader agent security platform. The branded product path becomes stronger when bundled with lifecycle evidence, runtime review, data masking, RBAC, and monitoring.
