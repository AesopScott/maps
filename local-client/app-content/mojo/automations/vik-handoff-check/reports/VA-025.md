Completed VA-025: Scoped API keys and agent secrets management

Claude's research output:
- Agent runtimes should isolate secrets from agent code, LLM context, persistent memory, logs, and debug traces.
- Secrets should be bound to capability scopes: specific endpoints, actions, tenants, workspaces, rate limits, time windows, and approval conditions.
- Runtime should use request-scoped injection: secrets are materialized only inside the controlled transport layer at outbound API-call time, never inside prompts or agent reasoning.
- Use opaque secret handles rather than raw secret values. Handles should support rotation, revocation, TTLs, and audit metadata without revealing secrets.
- Agents should access only a `SecretsClient` or equivalent stub; direct HTTP clients and direct secret reads should be blocked where possible.
- Guardrails should include prompt/memory opacity, redacted traces, curated transport, no wildcard scopes, immediate revocation, scope drift detection, behavioral baseline alerts, and external audit hooks.
- Proof should include prompt-injection leakage tests, memory-exfiltration tests, revoked-secret failure tests, rotation transparency tests, audit-log reconstruction, and compliance mapping for SOC2/HIPAA/PCI-style controls.

Vik architecture-owner interpretation:
VA-025 is a runtime contract, not a documentation footnote. Mojo agents should never reason over raw secrets. The correct boundary is a controlled transport/service layer that can say yes/no to a tool call, attach the credential, log the decision, and revoke access without the agent ever seeing the secret. This also clarifies authority: access to a tool is not access to every secret that tool could use. Secrets must be bound to a specific action, scope, tenant, and runtime context.

Productization recommendation:
Build as an internal enabling component first, then expose as SDK/API infrastructure if internal proof is strong. I do not recommend a standalone secrets-management product because the space is crowded and mature. I do recommend building agent-scoped secret handling as a mandatory part of Mojo's agent runtime and later making it visible as an enterprise trust feature: scoped secrets for agentic systems. It could become part of a broader Mindshare/Mojo Agent Platform, especially when bundled with authority binding, masking, audit, and monitoring.
