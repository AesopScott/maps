Completed VA-013: Attack surface discovery agents

Claude research output: attack surface discovery agents should use a three-layer scope-validation gate: declaration, instrumentation, and checkpoint. The declaration layer defines target zones, depth limits, method restrictions, finding categories, risk ceilings, and time/resource budgets. The instrumentation layer validates every discovery callsite before target queries, after findings, and on new traversal hops. The checkpoint layer validates staged results and audit logs before returning findings.

Architecture pattern:
- Scope Declaration Contract defines allowed domains, CIDRs, repos, cloud accounts, asset classes, methods, categories, depth, and time budget.
- Pre-flight Validation checks every target against the approved allowlist before probing or reading.
- Method Envelope enforces passive-only versus active mode and blocks unapproved protocol/action classes.
- Breadth and Depth Guards prevent graph expansion beyond declared traversal limits.
- Finding Filter strips or rejects results outside declared finding categories.
- Checkpoint Review validates all findings against scope before output.
- Immutable Audit logs every boundary decision, rejection, and attempted scope change.

Guardrails:
- Whitelist-only targets; no inferred scope.
- Fail closed when target or category is uncertain.
- No mid-run scope expansion without human approval.
- Passive-only mode disables active scans, probes, or service-impacting methods.
- Bounded traversal prevents discovery explosion.
- Findings outside category/scope are quarantined, not returned.
- Audit trail must prove which checks allowed or rejected each target/finding.

Eval suite:
- In-scope discovery succeeds for approved targets.
- Out-of-scope targets are rejected and logged.
- Depth limits block transitive expansion.
- Passive-only mode blocks active techniques.
- Mid-run scope redefinition is blocked.
- Post-filter catches indirect out-of-scope findings.
- Boundary violation prevention target is 100% before exposure.
- Audit replay reconstructs all boundary decisions.

My architecture-owner interpretation: VA-013 is a core trust gate for any discovery product. The failure mode is not only unauthorized scanning; it is unauthorized knowledge acquisition. A discovery agent must not expand from “inspect this asset” to “map the customer’s world” by convenience. This pattern directly supports project isolation, cloud/account discovery, exposure graph inputs, and future attack-surface products.

Productization recommendation: enabling component with product leverage. Do not sell the gate alone. Build it as reusable scope-validation infrastructure inside discovery, exposure graph, vulnerability validation, and cloud-security products. It becomes customer-visible proof: every finding can show it was collected inside approved scope. If branded later, frame it as part of a “safe discovery” or “audited exposure management” platform, not as the standalone object.

Boundaries held: no implementation beyond report/state/backlog record, no Git/release, production, external communication, spending, secrets, authority expansion, runtime activation, or promotion.
