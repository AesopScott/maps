Completed VA-028: Monitoring, response, and behavior drift for AI agents

Claude's research output:
- Use a layered detection model: signal collection, baseline profiling, anomaly detection, and response loop.
- Signals should include token budgets, spend patterns, tool-call sequences, latency, context-window use, memory transitions, prompt-injection attempts, output entropy, output length, semantic drift, and resource exhaustion.
- Baselines should be agent-specific, task-specific, and user-cohort-specific, covering expected tool use, token burn, errors, output format, and semantic behavior.
- Anomaly detection should use multiple signals: statistical outliers, behavioral deviation, intent mismatch, and weighted composite scoring. Single metrics should not create high-severity alerts by themselves.
- Response should escalate by confidence: log, rate-limit, pause, human review, state snapshot, and forensic preservation.
- Guardrails should include contextual allowlists, ensemble detection, gradual degradation, clean-run baseline updates, false-positive tuning, and clear suppression rules for known high-variance work.
- Proof should include synthetic red-team attacks, historical log replay, low-risk A/B scoring, detection-latency metrics, false-negative checks, false-positive analysis, and threshold ROC tuning.

Vik architecture-owner interpretation:
VA-028 should become Mojo's behavior-observation pattern for role agents and customer-facing security agents. The key is avoiding both extremes: silent drift and noisy panic. Agents need baseline-aware monitoring that can distinguish normal task expansion from policy bypass, prompt injection, tool abuse, memory corruption, or runaway cost. For Mojo, this connects heartbeat/file-watch discipline, observation logs, runtime hardening, and incident routing into one agent behavior model.

Productization recommendation:
Build as an internal capability now and an enabling component later. I would not sell this first as a standalone product; behavior monitoring is expected inside any trustworthy agent platform. But once proven across internal agents, it can become a strong product surface inside a broader agent security platform: anomaly scoring, drift dashboards, incident snapshots, and audit exports. Productization should wait until internal agents generate enough real baseline and incident data to prove low false positives.
