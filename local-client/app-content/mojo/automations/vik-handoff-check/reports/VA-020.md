Completed VA-020: Security knowledge and compliance mapping agents

Claude research output: security knowledge and compliance agents need a version-aware Knowledge Manifest. Each agent should know exactly which policy, RAG corpus, and current-state snapshot it is using, with manifest ID, effective date, expiration, checksum, source lineage, approval signatures, and audit trail. The manifest should be checked before decisions, logged with every decision, and rejected when expired, unsigned, corrupted, or inconsistent with loaded knowledge.

Architecture pattern:
- Knowledge Manifest declares policy layer, RAG layer, and state layer.
- Agents verify checksums before using knowledge.
- Every decision records manifest ID, policy version, corpus version, state snapshot, and timestamp.
- Mid-action policy or state changes trigger abort/retry, not silent continuation.
- Manifests are signed by approved policy/compliance owners.
- Agents cannot load untracked knowledge sources.
- Audit logs are append-only and tamper-evident.
- Knowledge access is role-scoped.

Guardrails:
- Unsigned or expired manifests block agent execution.
- RAG must come from approved corpora, not arbitrary web or untracked files.
- Policy knowledge must come from versioned source or approved policy system.
- State snapshots must declare age and source.
- Decisions made near stale boundaries should warn or halt depending on risk.
- Manifest expiration cannot be waived without a newly approved manifest.
- Every compliance/security decision must be replayable from manifest lineage.

Eval suite:
- Provenance: 100% of decisions trace to a manifest.
- Integrity: corrupted policy/RAG/state checksum is always rejected.
- Staleness: expired manifests block decisions.
- Audit completeness: random decisions can be reconstructed from logs.
- Conflict handling: mid-execution manifest updates cause abort/retry.
- Knowledge accuracy: expert review confirms decisions align with policy intent.
- Multi-agent scale: several agents share the same manifest and update consistently.
- Auditor proof: external compliance review can verify lineage and decision evidence.

My architecture-owner interpretation: VA-020 is the knowledge trust layer for every serious agentic security product. It answers the core customer question: “How do I know this agent made a decision from the right policy, right evidence, and right current state?” Without this, RAG-backed compliance and security agents become unverifiable. With it, Mojo can make stronger claims about auditability, policy freshness, and decision replay.

Productization recommendation: enabling component. Do not sell “version-aware knowledge” as a standalone SKU. Build it as shared infrastructure inside Mojo/Mindshare security products, especially analyst copilot, compliance mapping, exposure graph, cloud recommendation, and role/authority control-plane tools. It should eventually surface through customer-facing audit UI, but the product value is embedded trust, not the manifest itself.

Boundaries held: no implementation beyond report/state/backlog record, no Git/release, production, external communication, spending, secrets, authority expansion, runtime activation, or promotion.
