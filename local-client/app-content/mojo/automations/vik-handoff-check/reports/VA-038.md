Completed VA-038: Security data layer

Claude's research output:
- Security-agent data should normalize assets, vulnerabilities, threats, logs, and SIEM alerts into a common agent-consumable schema.
- Asset records need canonical identifiers, criticality, environment, owner, OS, and inventory metadata.
- Vulnerability records need CVE/identifier, severity, affected assets, remediation state, and SLA tier.
- Threat records need IOC/signature, source reputation, TLP/classification, and entity relationships.
- Logs and SIEM alerts need timestamps, event types, source/destination entities, actions, outcomes, rule IDs, correlation context, risk scores, and triage status.
- Provenance must be first-class: source system, ingestion timestamp, freshness SLA, transformation rules, confidence per field, and audit trail for agent decisions.
- Guardrails should include data classification, minimal-scope service principals, redaction of sensitive fields, separation of immutable reference data from mutable incident state, ingestion signatures, schema version gates, and evidence citation requirements.
- Proof should include source normalization pilots, real-incident validation, agent pilots, false-positive targets, query-latency targets, field-completeness metrics, no secret exposure, and schema extensibility tests.

Vik architecture-owner interpretation:
VA-038 is the data foundation for every higher-level security agent. Agents cannot reason reliably over unnormalized, stale, source-ambiguous data. Mojo needs a provenance-preserving security data layer where every entity, relationship, and alert carries freshness, source, confidence, and classification. This also prevents one of the main agent failure modes: confident recommendations built on stale or misjoined data.

Productization recommendation:
Build as an internal capability first, with possible branded product only after unique correlation logic emerges. I would not sell a generic normalization layer now; customers already have SIEMs, data lakes, and connector fatigue. The stronger path is to build a Mojo security data substrate for our agent products, then expose APIs/connectors if customers need integration. Product value comes from the combination of normalized data, evidence contracts, reasoning, and agent workflows, not schema alone.
