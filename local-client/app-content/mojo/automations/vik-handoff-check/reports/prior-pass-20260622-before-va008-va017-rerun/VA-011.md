Research output: rule tuning can quietly improve one detection path while creating blind spots elsewhere. An agent must not tune detection rules by local feedback alone. It needs regression proof across the full detection surface.

Architecture pattern:

`Propose Rule Change -> Run Baseline Regression -> Build Proof Matrix -> Gate Approval -> Shadow/Sandbox -> Promote or Reject -> Monitor Drift`

Three-layer validator pattern:

- Pre-deploy validator compares proposed rules against current baseline using historical signals, known incidents, near misses, synthetic cases, and negative controls.
- Rule-change proof matrix records reason, affected class, true/false positive and negative deltas, coverage delta, latency delta, cost delta, blind-spot delta, rollback plan, approval, and decision.
- Validator-regression policy rejects undocumented regressions and requires named approval for tradeoffs.

Allowed without approval:

- analyze rule performance
- propose candidate tuning
- run validation/eval
- produce proof matrix
- recommend promote, reject, or rollback

Blocked without approval:

- deploy rule changes live
- suppress a detection class
- materially lower severity threshold
- remove a noisy rule
- reduce logging or telemetry
- alter incident routing
- change production detection behavior

My architecture-owner interpretation: VA-011 should define a validator-regression policy and rule-change proof matrix. Agents may propose and validate rule tuning, but live rule promotion requires proof that no protected detection class regressed, or explicit named approval for a documented tradeoff.

Productization recommendation: internal capability first. Standalone detection-rule tuning has dangerous liability because a bad recommendation can create blind spots. Build regression-backed rule tuning internally, then consider bundling it inside a broader security product later.

Boundaries held: no implementation beyond this retroactive proof artifact, no Git/release, production, external communication, spending, secrets, authority expansion, runtime activation, or promotion.
