Research output: discovery agents need declarative scope plus runtime validation. Approved scope should be stored as a manifest with repo/path boundaries, file-type filters, sensitivity level, data classification, and permission limits. A middleware gate checks every discovery request before access. Enforcement should be deny-by-default, fail-secure on ambiguity, normalize paths before checking, apply uniformly to file reads/search/API calls, and audit both allowed and denied checks. Scope cannot be modified by the agent at runtime.

Architecture pattern:

`User/task request -> Scope manifest -> Request normalization -> Scope gate -> Discovery tool access -> Audit trace`

Scope manifest contract:

```text
agent_id
task_id
approved_roots
approved_repos
approved_globs
denied_globs
denied_paths
allowed_tool_classes
data_classification_allowed
sensitivity_ceiling
expiration
approver
audit_required
```

Required gate behavior:

- Deny by default.
- Resolve absolute/canonical path before check.
- Block traversal and symlink escape.
- Treat ambiguous paths as denied.
- Apply the same check to file read, list, search, grep, API fetch, RAG retrieval, and generated follow-up access.
- Keep scope immutable during a run.
- Require new approval for a new root, repo, or project.
- Log all passes and denials.

My architecture-owner interpretation: VA-013 should define a scope-validation gate for project/file discovery. Discovery agents may only inspect resources explicitly covered by an approved immutable scope manifest. Any ambiguous, inferred, sibling, secret, production, external, or cross-project access is denied and escalated for re-authorization.

Productization recommendation: enabling component. Standalone attack-surface management is crowded and operationally risky. Build the scope-validation discovery engine first, likely feeding VA-015 exposure relationship analysis.
