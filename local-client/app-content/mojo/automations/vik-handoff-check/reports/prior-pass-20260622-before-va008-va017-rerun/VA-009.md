Completed VA-009: AI malware reverse-engineering assistant

Claude research output: AI-assisted malware analysis should use a safe-static-analysis-only architecture. The assistant classifies intent, accepts defensive analysis requests, rejects weaponization/evasion requests, parses artifacts without execution, and outputs threat understanding, detection ideas, confidence tags, and mitigation guidance. It must not execute samples, generate payloads, provide evasion guidance, or help build attack infrastructure.

Architecture pattern:
- Input Validation and Intent Classification separates defensive analysis from weaponization or attack-development requests.
- Static Analysis Pipeline permits disassembly/decompiled-structure review, AST/control/data-flow inspection, entropy/header/signature metadata, string extraction, IOC matching, MITRE mapping, and static correlation.
- Controlled Output provides threat assessment, detection strategy, mitigation path, and confidence distinction between observed evidence and inference.
- Hard stop on dynamic execution, sandboxing requests, runtime behavior simulation, process monitoring, payload generation, persistence, C2 setup, or detection bypass.

Guardrails:
- No code execution, interpretation, compilation, or sample detonation.
- No exploit scaffolding, payload stubs, evasion logic, persistence instructions, or attack infrastructure guidance.
- Detection logic is allowed; detection-bypass logic is blocked.
- Malware behavior explanation is allowed; behavior replication instructions are blocked.
- Audit every query and response.
- Borderline requests require manual review.
- Rate-limit and flag repeated evasion-oriented queries.
- Quarterly adversarial prompt sweeps must test guardrail gaps.

Eval suite:
- Injection and jailbreak attempts must fail to produce payload, evasion, or infrastructure guidance.
- Misuse prompts must receive refusal and safe redirection.
- Static-analysis accuracy against de-identified malware samples.
- Time-to-insight compared to analyst baseline.
- Confidence calibration for high-confidence versus inferred findings.
- Guardrail robustness across sustained query volume.
- False-negative rate for block-worthy requests.
- Manual-review alert quality so security ops is not overwhelmed.

My architecture-owner interpretation: VA-009 belongs near malware triage and safe script/hook review. The main product risk is dual use: a tool that explains malware can accidentally become a tool for improving malware. Mojo’s safe lane is static analysis, defensive explanation, detection support, and mitigation. Any dynamic analysis or exploit replication belongs in a separate lab-only workflow with explicit approval, not in this assistant.

Productization recommendation: enabling component. Do not launch as a standalone malware-analysis product now. The market is specialized, incumbent-heavy, and liability-prone. Build internally as safe-static-analysis policy and analyst-assist module. Later embed as a co-analyst feature inside threat workbench, SOC copilot, secure code review, or partner security tooling if accuracy, safety, legal, and customer demand are proven.

Boundaries held: no implementation beyond report/state/backlog record, no Git/release, production, external communication, spending, secrets, authority expansion, runtime activation, or promotion.
