Research output: ransomware response needs fast detection, but containment actions are themselves high-risk destructive actions. Detection and escalation can be automated. Containment must remain approval-gated.

Architecture pattern:

`Detect -> Classify -> Evidence Packet -> Escalate -> Approval Gate -> Scoped Containment`

Detection classes:

- Volume anomaly: mass delete, mass overwrite, recursive wipe, abnormal mutation rate.
- Entropy/encryption pattern: sudden extension churn, randomized filenames, high-entropy rewrites.
- Scope expansion: same actor/process touching many unrelated directories, shares, repos, users, or hosts.
- Backup/recovery risk: backup deletion, restore-point deletion, retention change, backup job disablement.
- Identity/control risk: admin escalation, service-account misuse, mass permission change, MFA/security weakening.
- Observability tampering: logging disabled, alerts muted, endpoint agent stopped.
- Destructive command intent: requested action would wipe, quarantine, revoke, isolate, rotate, delete, or disable critical state.

Allowed without containment approval:

- inspect approved telemetry
- classify severity
- pause the agent's own workflow
- build evidence packet
- notify approved owner
- recommend containment options

Blocked without explicit approval:

- isolate host/network
- kill process
- revoke account/session
- rotate secrets
- delete/quarantine files
- restore backups
- change firewall/DNS/cloud config
- touch production, Git/release, or external communications

My architecture-owner interpretation: VA-008 should become a destructive-action guardrail requirement. Agents may auto-detect and auto-escalate urgent destructive risk, but may not execute containment unless a named human or preapproved playbook grants one scoped, auditable action.

Productization recommendation: internal capability first. High value, high liability. Do not launch standalone early. Build internally as a destructive-risk sentinel plus approval-gated containment pattern; consider productization only after proof that false positives and containment risk are controlled.

Boundaries held: no implementation, file edits beyond this retroactive proof artifact, Git/release, production, external communication, spending, secrets, authority expansion, runtime activation, or promotion.
