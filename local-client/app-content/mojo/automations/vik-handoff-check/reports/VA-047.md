Completed VA-047: Attack path mapping for AI security agents

Claude's research output:
- Architecture: represent attack paths as immutable declarative graphs, not executable plans.
- Graph model: nodes for assets, vulnerabilities, controls, and trust boundaries; edges for exploit dependencies, privilege escalation chains, detection surface, and actor context.
- Edge metadata: required privileges, exploitability confidence, remediation dependencies, time-to-compromise, detection methods, patch availability, and control decay.
- Guardrails: mandatory intent declaration, graph-only access, no network/privilege/control modification capability, output filtering, reward constraints against exploit execution, and forensic query logging.
- Safe outputs: risk rankings, remediation sequencing, control coverage gaps, incident path reconstruction, and defensive analysis. Avoid exploit instructions, shellcode, exfiltration methods, or offensive reconnaissance guidance.
- Eval/proof: adversarial red-team jailbreak attempts, capability baseline, instrumented constraint tests, refusal logging, and benchmark against human defensive analysis.
- Product recommendation from Claude: build as enabling component with product path, not narrow standalone product initially. Potential later Threat Navigator agent.

Vik's architecture-owner interpretation:
- Attack path mapping is powerful because it compresses environment relationships into actionable defensive prioritization. It is also risky because the same graph can guide offense if output and tool boundaries are weak.
- The architecture must enforce graph-as-evidence, not graph-as-execution. AI agents may read and reason over paths, but cannot perform scanning, exploitation, lateral movement, or control changes under this item.
- The strongest product primitive is a Defensive Attack Path Graph: business-impact-ranked paths, evidence sources, control gaps, remediation options, and approval-gated handoffs.
- Safe use depends on intent classification, output redaction, tenant isolation, audit trail, and refusal evals for offensive prompts.

Productization recommendation:
- Recommendation: recommend_make as an enabling security product component; recommend_further_review before standalone agent launch.
- Why: attack path mapping has high product value for risk scoring, vulnerability prioritization, architecture review, and SOC context. Standalone autonomous attack-path agent needs stronger abuse-prevention and customer proof.
- Product path: build internal Defensive Attack Path Graph spec first. Later expose as product capability in Vulnerability Prioritization Agent or Security Decision Engine; consider branded Threat Navigator only after red-team proof.
