Completed VA-037: Security reasoning layer

Claude's research output:
- Security reasoning should be a structured inference engine with explicit evidence binding.
- Core components should include normalization, signal extraction, correlation, attack-path inference, risk quantification, decision engine, and evidence ledger.
- Inputs should normalize SIEM alerts, threat feeds, behavioral signals, asset discovery, and other security data into a provenance-aware schema with confidence metadata.
- Correlation should cluster signals by time, asset, actor, and pattern; map narratives to kill-chain phases and MITRE ATT&CK-style techniques where useful.
- Risk scoring should combine asset criticality, exposure, blast radius, signal confidence, and inferred attacker capability.
- Each recommendation should be a claim backed by an evidence contract: signals, sources, time windows, confidence, inference path, recommendation, action option, and audit record.
- Guardrails should require explainability, adversarial robustness against log poisoning and alert flooding, confidence tiers, analyst override, feedback loops, and versioned audit trails.
- Proof should include known incident reconstruction, poisoned-alert tests, precision/recall, analyst acceptance, time-to-recommendation, and outcome tracking.

Vik architecture-owner interpretation:
VA-037 is the reasoning core for security-agent products. The critical boundary is that the agent may recommend only what the evidence contract supports. Reasoning must be inspectable: which facts, from which sources, at which confidence, led to which recommendation. This pattern fits Mojo's broader control plane: no unsupported urgency, no black-box risk scores, no action without evidence, and no authority leap from inference to remediation.

Productization recommendation:
Build as an internal capability first, with optional partner/API productization after proof. I would not immediately sell a generic reasoning engine because it depends heavily on customer data quality, asset context, and SOC trust. Internally, it can become a major advantage for security tools and managed services. Productization should wait until the evidence contract, analyst acceptance, and false-positive metrics are strong enough to survive customer scrutiny.
